# 行為面試

> 技術再強，行為面試搞砸一樣拿不到 offer。**行為面試（behavioral interview）** 考的不是你會不會寫程式，而是「你是怎樣的工程師、怎樣的同事」——如何面對衝突、失敗、壓力、協作。這章講行為面試的邏輯、**STAR 回答框架**，以及常見問題的準備。

## 💡 白話導讀（建議先讀）

技術面過了,行為面掛掉——比你想的常見。先懂面試官的邏輯:
**過去的行為是未來的最佳預測**,所以他們只問一種問題:
「**講一次你……的真實經驗**」(處理衝突、扛下失誤、推動困難的事)。

他們要聽的是**故事,不是形容詞**。
「我很有責任感」零分——誰都會說;
「上線事故那晚我做了什麼」才有分數。而說故事有公認的模板——**STAR**:

- **S**ituation:背景一兩句(什麼專案、什麼壓力)。
- **T**ask:你的任務與難處(注意是「**我**」不是「我們」——
  面試官要評的是你,不是你的團隊)。
- **A**ction:你**具體做了什麼**——故事的主體,講決策與取捨,不只講結果。
- **R**esult:成果+數字(「p99 從 800ms 降到 120ms」)+**學到什麼**
  (反思是資深訊號)。

準備方法很機械,照做就好:**挑 5~6 個真實專案故事,每個都能switch著回答
不同問題**(同一個事故故事,可以答「處理壓力」也能答「跨團隊溝通」)。
面試前把每個故事按 STAR 寫成小抄、講給自己聽三遍。

這章附高頻題庫(失敗經驗、意見衝突、時程不可能、帶人/被帶)、
每題的「面試官在聽什麼」、以及台灣/外商常見的追問模式。

## Why（為什麼）

很多工程師把所有準備時間投在刷演算法題，卻在行為面試翻車——因為他們以為「技術好就夠了」。但公司雇的是**人**，不只是**寫程式的機器**。行為面試回答的是技術面試回答不了的問題：

- **你好不好共事？** 團隊要跟你朝夕相處，你會不會製造衝突、能不能溝通。
- **你怎麼面對困難？** 專案失敗、deadline 壓力、意見不合時，你的反應。
- **你有沒有 ownership？** 遇到問題是扛起來解決，還是推責任。
- **你的成長性與價值觀** 是否符合團隊文化。

行為面試常是**決定性**的一關——尤其資深職位，技術到一定水準後，「軟實力」往往是拉開差距的關鍵。而它其實**可以準備**：問題類型是可預期的（衝突、失敗、領導、影響力…），只要用對框架、備好真實故事，就能穩定發揮。這章講清楚怎麼準備，讓你的技術實力不被行為面試拖累。它與整本書的技術內容互補——技術讓你能做事，行為面試展示你怎麼做事、和誰一起做事。

## Theory（理論：行為面試的底層邏輯）

**行為面試的核心假設**：**過去的行為，是未來行為的最佳預測**。所以面試官問的都是「**講一次你……的經驗**」——過去實際發生的事，而非假設性的「你會怎麼做」。他們想聽**具體的真實故事**，不是空泛的自我評價（「我很有責任感」誰都會說，講一個「你扛下責任」的故事才有說服力）。

面試官在故事裡聽什麼（**評分維度**）：

- **你的角色與貢獻**：你**具體做了什麼**（不是「我們團隊」，是「**我**」）。
- **面對困難的方式**：分析、決策、行動、溝通。
- **結果與反思**：帶來什麼成果、學到什麼。
- **軟實力訊號**：溝通、協作、領導、ownership、成長心態、同理心。

**常見問題類型**（幾乎所有行為問題的變體）：

- **衝突**：與同事/主管意見不合怎麼處理。
- **失敗/錯誤**：講一次你失敗/犯錯的經驗，怎麼收拾、學到什麼。
- **挑戰/困難**：最難的技術/專案挑戰。
- **領導/影響力**：帶領或推動一件事、影響他人。
- **優先級/壓力**：deadline 壓力、多任務取捨。
- **協作**：跨團隊合作、幫助他人。

## Specification（規範：STAR 框架）

**STAR** 是結構化回答行為問題的黃金框架，讓你的故事清楚、完整、有重點：

- **S — Situation（情境）**：簡述背景。是什麼專案、什麼處境？（簡潔，1-2 句，給足脈絡即可）。
- **T — Task（任務）**：你要達成什麼目標、面對什麼問題、你的責任是什麼？
- **A — Action（行動）**：**你**具體採取了哪些行動？（**這是重點，要最多篇幅**——講清楚你的思考、決策、做的事，強調「我」而非「我們」）。
- **R — Result（結果）**：結果如何？最好**量化**（提升 X%、省下 Y 小時、解決了 Z）。加上**反思/學到什麼**更佳。

**準備方法——備好一組「故事庫」**：

1. 回顧你的經歷，挑出 **5-8 個豐富的真實故事**（有挑戰、有你的行動、有結果）。
2. 每個故事用 STAR 寫成筆記。
3. 一個好故事常能**覆蓋多種問題**（一個「解決線上事故」的故事可回答「壓力」「ownership」「技術挑戰」「協作」）。
4. 練習講出來——控制在 2-3 分鐘，流暢自然。

## Implementation（怎麼講得好：細節與心法）

**為何要具體、量化**：面試官每天聽無數空泛回答。「我改善了系統效能」蒼白無力；「我用 cProfile 定位到一個 N+1 查詢，改成 eager loading，把 API p99 從 800ms 降到 120ms」立刻可信、有份量——**具體的技術細節 + 量化結果**是可信度的來源。這也是為何 STAR 的 Action 要展開講：細節證明「這是你真的做過的事、你真的懂」。

**為何強調「我」而非「我們」**：團隊合作值得肯定，但面試官要評估的是**你個人**的能力與貢獻。整段都說「我們決定、我們做了」，面試官無法判斷**你**做了什麼。要清楚區分「團隊的脈絡（我們）」和「**我的**具體行動（我提議、我實作、我協調）」。

**負面問題（失敗/衝突）怎麼答**：這類問題**不是陷阱，是機會**——面試官想看你的**成熟度與成長**。訣竅：

- **選真實但已妥善處理的故事**（別選毀滅性、或你完全是受害者/完全甩鍋的）。
- **誠實承認自己的責任**（不推給別人、不裝完美）——展現 ownership 與自省。
- **重點放在「你怎麼處理 + 學到什麼 + 之後如何改進」**——把失敗轉成成長的證明。
- 衝突題強調**同理、溝通、以資料/共同目標化解**，而非「我贏了」。

**誠實原則**：**別編故事**。有經驗的面試官會追問細節（「那時你具體怎麼做的？」「為什麼選這個方案？」），編的故事經不起深挖。用**真實**的經歷，才能自然應對追問。

## Code Example（範例：用 STAR 拆解一個回答）

行為面試沒有程式碼，但我們可以用一個「STAR 結構檢查器」示範如何組織回答（純標準庫，可執行）：

```python
# star_helper.py — 用 STAR 框架組織行為面試回答（純標準庫，可執行）
from __future__ import annotations

from dataclasses import dataclass


@dataclass
class StarAnswer:
    situation: str
    task: str
    action: str
    result: str

    def is_complete(self) -> bool:
        """四要素齊全才是完整的 STAR 回答。"""
        return all([self.situation, self.task, self.action, self.result])

    def format(self) -> str:
        return (
            f"[情境] {self.situation}\n"
            f"[任務] {self.task}\n"
            f"[行動] {self.action}\n"
            f"[結果] {self.result}"
        )


def main() -> None:
    # 範例：回答「講一次你解決棘手技術問題的經驗」
    answer = StarAnswer(
        situation="某次大促，訂單 API 在尖峰時 p99 延遲飆到 800ms、頻繁逾時。",
        task="我負責找出瓶頸並在下次大促前解決，目標把 p99 壓回 200ms 內。",
        action=(
            "我用 cProfile 剖析，定位到訂單列表有 N+1 查詢；"
            "改成 eager loading 合併查詢、加上 Redis 快取熱門商品、"
            "並補了壓力測試驗證。過程中和 DBA 協調索引調整。"
        ),
        result="p99 從 800ms 降到 120ms，大促零逾時；我把這套排查流程寫成文件供團隊複用。",
    )
    print("STAR 回答:")
    print(answer.format())
    print(f"\n四要素完整: {answer.is_complete()}")

    # 反例：只有情境、缺具體行動與結果 → 空泛
    weak = StarAnswer(
        situation="系統很慢。", task="", action="我優化了一下。", result=""
    )
    print(f"\n空泛回答四要素完整: {weak.is_complete()}  ← 缺任務與結果，沒說服力")


if __name__ == "__main__":
    main()
```

**預期輸出**：

```pycon
$ python star_helper.py
STAR 回答:
[情境] 某次大促，訂單 API 在尖峰時 p99 延遲飆到 800ms、頻繁逾時。
[任務] 我負責找出瓶頸並在下次大促前解決，目標把 p99 壓回 200ms 內。
[行動] 我用 cProfile 剖析，定位到訂單列表有 N+1 查詢；改成 eager loading 合併查詢、加上 Redis 快取熱門商品、並補了壓力測試驗證。過程中和 DBA 協調索引調整。
[結果] p99 從 800ms 降到 120ms，大促零逾時；我把這套排查流程寫成文件供團隊複用。

四要素完整: True

空泛回答四要素完整: False  ← 缺任務與結果，沒說服力
```

這個範例展示了一個好 STAR 回答的要素：**具體情境、明確任務、詳實的個人行動（含技術細節，呼應本書 [profiling](../18-performance/01-profiling.md)、[N+1](../15-database/20-n-plus-1.md)、[快取](../15-database/18-redis.md)）、量化結果 + 複用價值**。對比空泛回答（缺任務與結果），高下立判。

## Best Practice（最佳實踐）

- **用 STAR 框架組織每個回答**：情境→任務→行動→結果，重點在「行動」。
- **準備 5-8 個真實故事**，涵蓋衝突/失敗/挑戰/領導/協作，一個故事覆蓋多題。
- **強調「我」的具體貢獻**，並用技術細節與量化結果增加可信度。
- **負面問題誠實承認責任、聚焦處理與成長**：把失敗講成學習。
- **絕不編故事**：面試官會追問，真實經歷才經得起深挖。
- **練習口說**，控制在 2-3 分鐘，流暢不背稿。
- **研究公司價值觀**，選能呼應其文化的故事。
- **準備問面試官的問題**：展現你對團隊/工作的真誠興趣。

## Common Mistakes（常見誤解）

- **只準備技術、忽略行為面試**：資深職位常在這關被刷。
- **回答空泛、無具體例子**：「我很有責任感」沒有故事支撐 = 沒說服力。
- **通篇「我們」不講「我」**：面試官無法評估你個人的貢獻。
- **行動講太少、情境講太多**：STAR 的重點是 Action，別把時間花在鋪陳背景。
- **負面問題裝完美或全甩鍋**：顯得不自省、無 ownership。
- **編造/誇大故事**：追問就露餡，信任崩塌。
- **結果不量化**：「改善了效能」遠不如「p99 降 85%」。
- **背稿般僵硬**：不自然、無法應對追問。

## Interview Notes（面試重點）

- **理解行為面試的邏輯**：過去行為預測未來，面試官要具體真實故事而非空泛評價。
- **熟練 STAR 框架**：情境/任務/行動/結果，重點在個人行動與量化結果。
- **備好故事庫**（5-8 個），涵蓋衝突、失敗、挑戰、領導、協作，一事多用。
- **負面問題**：誠實、擔責、聚焦成長，別裝完美或甩鍋。
- **強調「我」、用細節與數字**，並保持誠實（經得起追問）。
- **把技術經歷轉成好故事**：本書的排查、優化、架構決策都是絕佳素材。
